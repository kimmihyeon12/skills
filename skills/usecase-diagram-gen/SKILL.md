# usecase-diagram-gen

기획 문서, RFP, 메모 등 어떤 형태의 원본 문서라도 받아서 **도메인 경계 기반 유즈케이스 다이어그램**을 Mermaid 형식의 MD 파일로 만든다. Mermaid에는 UML 유즈케이스 다이어그램 타입이 없으므로 `flowchart` + `subgraph` 조합으로 근사한다.

---

## 지원 입력 형식

| 형식 | 처리 방식 |
|------|-----------|
| `.pdf` | 내장 Node.js 스크립트로 텍스트 추출 후 분석 |
| `.txt` | 직접 읽어서 분석 |
| `.md`  | 직접 읽어서 분석 |

그 외 파일 형식은 지원하지 않는다. 사용자에게 위 세 가지 중 하나로 변환해달라고 안내한다.

---

## 실행 절차

### Step 0 — 모드 감지

실행 시작 시 워크스페이스 루트에 `usecase/` 폴더가 이미 존재하는지 확인한다.

- **`usecase/` 없음** → **신규 생성 모드**: Step 1부터 순서대로 진행
- **`usecase/` 있음** → **업데이트 모드**: 아래 [업데이트 모드 상세 절차](#업데이트-모드-상세-절차)로 건너뜀

### Step 1 — 파일 확인

사용자가 제공한 파일 경로의 확장자를 확인한다.
- `.pdf/.txt/.md` 이외 → 명확한 오류 메시지와 함께 종료
- 파일이 존재하지 않으면 → 경로 재확인 요청

### Step 2 — 텍스트 추출

**PDF 입력일 때:**

```bash
# 의존성 설치 (최초 1회)
cd ~/.claude/skills/usecase-diagram-gen/scripts && npm install --silent

# 텍스트 추출 — 워크스페이스 루트 기준 경로로 저장
node ~/.claude/skills/usecase-diagram-gen/scripts/extract_pdf_text.js \
  <입력.pdf> -o <워크스페이스_루트>/usecase/extracted/source.txt
```

추출된 파일은 **반드시 `usecase/extracted/source.txt`** 경로에 저장한다. 이후 Read 도구로 읽어 분석에 사용한다.

**TXT/MD 입력일 때:** Read 도구로 직접 읽는다.

### Step 3 — 도메인 분석

텍스트에서 아래 항목을 식별한다:

1. **액터(Actor)**: 시스템을 사용하는 역할 (예: 일반 사용자, 관리자, 외부 시스템)
2. **도메인(Domain)**: 기능적으로 유사한 유즈케이스의 집합 (예: 인증, 결제, 콘텐츠 관리)
3. **유즈케이스(Use Case)**: 각 도메인 내 구체적인 사용자 행동

도메인은 4-8개 수준으로 나누는 것이 가독성에 좋다. 너무 세분화하거나 하나로 합치지 않도록 균형을 잡는다.

도메인 목록이 확정되면 **도메인 간 의존성/선행 순서**를 결정한다. 이 순서가 파일 넘버링에 반영된다.

순서 판단 기준:
- 다른 도메인 대부분이 전제로 삼는 도메인이 앞 (예: 인증, 공통 시스템)
- 비즈니스 흐름상 먼저 실행되는 도메인이 앞 (예: 계획 → 실행 → 정산)
- 공통·기반·보안 성격의 도메인은 앞 번호
- 외부 연동·통계·리포트 성격의 도메인은 뒤 번호

**3-A. 요구사항 ID 감지**: 원본 문서에 요구사항 자체 ID 패턴이 있는지 확인한다.
- 감지 패턴 예: `REQ-001`, `FR-01`, `F-001`, `UC-001`, `기능번호`, `요구사항번호` 등
- 패턴이 없으면 → 요구사항 ID 추적 생략, UC ID만 부여
- 패턴이 있으면 → 각 유즈케이스 도출 시 원본 요구사항 ID를 함께 기록

**3-B. UC ID 부여**: 도메인 순서 확정 후 각 유즈케이스에 고유 ID를 부여한다.

UC ID 체계: `UC-[도메인번호]-[순번]` (도메인 번호는 파일 넘버링과 동일, 순번은 도메인 내 01부터 시작)

분석 결과를 아래 형식으로 정리한 뒤 다이어그램 작성에 진입한다:
```
01. [도메인: 공통 시스템]
  - UC-01-01: 공통 코드 조회 ← REQ-001
  - UC-01-02: 파일 첨부 ← REQ-002

02. [도메인: 인증]
  - UC-02-01: 회원가입 ← REQ-010, REQ-011
  - UC-02-02: 로그인 ← REQ-012
  - UC-02-03: 비밀번호 찾기 ← REQ-013
```

### Step 4 — Mermaid 다이어그램 작성

`references/mermaid-usecase.md`를 참고해 다이어그램을 작성한다.

기본 구조 (UC ID 포함):
```mermaid
flowchart LR
    actor1[👤 일반 사용자]
    actor2[🔧 관리자]

    subgraph 인증
        uc1(("UC-02-01\n회원가입"))
        uc2(("UC-02-02\n로그인"))
        uc3(("UC-02-03\n비밀번호 찾기"))
    end

    subgraph 결제
        uc4(("UC-03-01\n결제 수단 등록"))
        uc5(("UC-03-02\n주문 결제"))
        uc6(("UC-03-03\n환불 요청"))
    end

    actor1 --> uc1
    actor1 --> uc2
    actor1 --> uc3
    actor1 --> uc4
    actor1 --> uc5
    actor2 --> uc6
```

- 전체 overview 다이어그램 1개 + 도메인별 상세 다이어그램 생성
- 도메인이 5개 이상이면 overview는 각 도메인의 핵심 유즈케이스 2-3개만 표시해 가독성 유지

### Step 5 — 결과물 저장

워크스페이스 루트(사용자가 현재 작업 중인 프로젝트 루트)에 아래 구조로 저장:

```
usecase/
├── README.md                          ← 도메인 목록, 액터 설명, 파일 링크
├── overview.md                        ← 전체 도메인 overview Mermaid 다이어그램
├── 01-common-system-usecase.md        ← 도메인별 상세 Mermaid 다이어그램 (Step 3 순서 반영)
├── 02-security-usecase.md
├── 03-data-management-usecase.md
├── ...
└── extracted/                         ← PDF 입력 시만 생성
    └── source.txt
```

**파일명 규칙:**
- 도메인 파일에는 반드시 2자리 zero-padding 번호를 접두어로 붙인다: `01-`, `02-`, ... `10-`, `11-`
- 번호는 Step 3에서 결정한 도메인 순서 그대로 부여한다
- 한국어 도메인명은 영문으로 변환한다 (예: `03-authentication-usecase.md`)
- `overview.md`와 `README.md`는 번호 없이 유지한다 (항상 최상위 진입점)
- `README.md`의 파일 링크도 넘버링된 파일명으로 작성한다

각 도메인 파일에는 다이어그램 아래에 반드시 추적 테이블을 포함한다:

**요구사항 ID가 있는 경우 — 요구사항 추적 테이블:**
```markdown
## 요구사항 추적

| UC ID | 유즈케이스 | 요구사항 ID | 비고 |
|-------|-----------|------------|------|
| UC-02-01 | 회원가입 | REQ-010, REQ-011 | |
| UC-02-02 | 로그인 | REQ-012 | |
```

**요구사항 ID가 없는 경우 — UC 목록 테이블:**
```markdown
## 유즈케이스 목록

| UC ID | 유즈케이스 | 설명 |
|-------|-----------|------|
| UC-02-01 | 회원가입 | 신규 사용자 이메일 등록 |
| UC-02-02 | 로그인 | 이메일/비밀번호 인증 |
```

저장 완료 후 생성된 파일 목록과 식별된 도메인/액터 요약을 사용자에게 보고한다.

## 업데이트 모드 상세 절차

워크스페이스에 이미 `usecase/` 폴더가 있을 때 사용하는 플로우다. 기존 다이어그램을 유지하면서 변경사항만 반영한다.

### U-Step 1 — 현황 파악

기존 파일을 모두 읽어 현재 상태를 파악한다:
- `usecase/README.md` — 도메인 목록, 액터 목록
- `usecase/overview.md` — 전체 구조
- `usecase/[도메인명]-usecase.md` — 각 도메인 상세

머릿속에 "현재 도메인 N개, 각 도메인별 유즈케이스 목록, 액터 목록"을 정리해둔다.

### U-Step 2 — 변경 내용 파악

사용자로부터 무엇이 바뀌었는지 파악한다:

- **새 문서가 있는 경우**: Step 1~3(파일 확인 → 텍스트 추출 → 도메인 분석)을 수행한 뒤, 기존 현황과 비교
- **구두 지시만 있는 경우**: 사용자 설명에서 직접 변경사항 도출

변경 유형을 분류한다:
```
[추가] 새 도메인, 새 유즈케이스, 새 액터
[수정] 유즈케이스 이름 변경, 관계 변경, 액터 연결 변경
[삭제] 유즈케이스 제거, 도메인 통합/제거
```

애매한 부분이 있으면 작업 전에 사용자에게 확인한다.

### U-Step 3 — 선택적 수정

**변경이 있는 파일만** 수정한다. 영향을 받지 않는 도메인 파일은 건드리지 않는다.

| 변경 유형 | 수정 대상 파일 |
|-----------|----------------|
| 특정 도메인 내 유즈케이스 추가/수정/삭제 | 해당 도메인 `.md` + `overview.md` (핵심 UC가 바뀐 경우) + `README.md` |
| 새 도메인 추가 | 새 도메인 `.md` 생성 + `overview.md` + `README.md` |
| 도메인 삭제/통합 | 해당 `.md` 삭제 + `overview.md` + `README.md` |
| 액터 추가/변경 | 영향받는 도메인 `.md` + `overview.md` + `README.md` |

**넘버링 처리:**
- 기존 도메인 파일의 번호는 원칙적으로 유지한다
- 새 도메인을 중간에 삽입해야 할 경우: 해당 위치 이후 파일들을 모두 한 번호씩 밀고 재조정한다
- 순서 무관하게 단순 추가라면 마지막 번호 다음을 이어서 붙인다
- 번호가 바뀐 파일은 `README.md` 링크도 함께 업데이트한다

**UC ID 처리:**
- 기존 UC ID는 절대 변경하지 않는다 (추적 이력 보존)
- 새 유즈케이스는 해당 도메인의 마지막 UC 순번 다음을 이어서 부여한다
- 새 도메인은 기존 마지막 도메인 번호 다음을 이어서 부여한다
- 삭제된 UC의 ID는 재사용하지 않는다

### U-Step 4 — 변경 요약 보고

작업 완료 후 사용자에게 변경 내용을 요약 보고한다:

```
✅ 업데이트 완료

[수정된 파일]
- authentication-usecase.md: 소셜 로그인 유즈케이스 추가
- overview.md: 결제 도메인 핵심 UC 반영

[추가된 파일]
- notification-usecase.md: 알림 도메인 신규 추가

[변경 없는 파일]
- content-usecase.md, user-management-usecase.md (변경 없음)
```
